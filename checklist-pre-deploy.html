Roadmap a Producción

ANTES DE LA
REESTRUCTURA

Tareas a completar antes de iniciar la migración a Astro + Cloudflare Pages + Cloudinary + Decap CMS. Los checks se guardan automáticamente en tu navegador.

01

Cuentas y Servicios

Crear las cuentas en los servicios que vamos a usar. Todas tienen free tier suficiente para empezar.

  • Cloudflare — hosting + DNS + analytics + email routing. cloudflare.com — registrarse con email del cliente si va a manejar él el dominio.
  • Cloudinary — hosting de fotos y videos optimizados. cloudinary.com/users/register/free — anotar: cloud_name, api_key, api_secret.
  • Web3Forms — backend del formulario de contacto. web3forms.com — pedir access_key indicando el email destinatario.
  • UptimeRobot — alertas si el sitio se cae. uptimerobot.com — configurar después del deploy.
  • GitHub — repo privado del proyecto (necesario para Decap CMS y deploy automático). Si el repo actual es local, crear repo en GitHub y hacer push.
02

Decisiones con el Cliente

Conversaciones a tener con el cliente antes de empezar a codear. Bloquean partes del desarrollo.

  • Dominio definitivo — ¿está comprado? ¿en qué registrador? Si todavía no está, comprarlo (Namecheap, Cloudflare Registrar o el que prefieras).
  • Email corporativo — definir solución (Zoho Mail free, Google Workspace pago, o forwarding gratis con Cloudflare Email Routing).
  • Email destinatario del formulario — a dónde llegan los mensajes que envían los clientes.
  • WhatsApp Business — ¿incluir botón flotante con link directo wa.me? Pedir número.
  • Textos SEO finales — title (máx 60 chars) y meta description (máx 155 chars) por cada página. Son lo que ve la gente en los resultados de Google. Aprobar con el cliente.
  • Palabras clave principales — qué quiere posicionar (ej: "productora audiovisual buenos aires", "videos para eventos").
  • Idiomas — ¿solo español, o también inglés? Define la estructura de rutas (/en/, /es/).
  • Política de privacidad y términos — necesarios por el formulario. Pedir al cliente o redactar versión genérica para que apruebe.
03

Contenido a Recopilar

Assets y textos que el cliente tiene que mandar antes de migrar. Sin esto el sitio nuevo arranca incompleto.

  • Logo en SVG — no PNG. Vectorial, escala perfecto en cualquier tamaño. Si solo hay PNG, pedirlo al diseñador.
  • Favicon set completo — 16x16, 32x32, 180x180 (apple-touch-icon), 192x192 y 512x512 (PWA), versión SVG. Generador rápido: realfavicongenerator.net
  • Open Graph image (1200x630px) — la imagen que aparece al compartir el link en WhatsApp, Instagram, Facebook. Una por página principal idealmente, o una sola para todo el sitio mínimo.
  • Bio del estudio — descripción larga para usar en schema.org y página "Nosotros".
  • Dirección física, teléfono, horarios — para SEO local y schema LocalBusiness (si aplica).
  • Links a redes sociales — Instagram, Behance, Vimeo, YouTube, LinkedIn, etc.
  • Listado definitivo de trabajos — qué mostrar en el portfolio: título, cliente, año, descripción, link a media.
  • Videos originales en máxima calidad — Cloudinary se encarga de optimizar. No mandes versiones ya comprimidas.
  • Fotos en alta resolución — mínimo 2000px del lado largo. Para hero/portfolio mejor 3000px+.
  • Bios del equipo — si la página "Nosotros" los menciona, foto + nombre + rol + bio corta.
04

Auditoría del Estado Actual

Asegurarse de que el código actual está limpio y respaldado antes de tocar nada.

  • Backup con tag git — crear tag v1-html-puro en el commit actual para poder volver si algo sale mal.
  • Inventario de assets actuales — listar fotos/videos en el repo y marcar: mantener / reemplazar / descartar.
  • Revisar info sensible — confirmar que no hay API keys, emails personales o credenciales committeadas. Usar herramienta: gitleaks o trufflehog.
  • Validar HTML actual — pasar cada página por validator.w3.org y corregir errores antes de migrar.
  • Accesibilidad básica — revisar contraste de colores, alt text en imágenes, navegación con teclado.
05

Performance Baseline

Medir el estado actual para después poder demostrar la mejora del sitio nuevo.

  • Lighthouse — correr en cada página del sitio actual y guardar los 4 scores (Performance, Accessibility, Best Practices, SEO). Chrome DevTools → Lighthouse → Generate report.
  • PageSpeed Insights — anotar los Core Web Vitals (LCP, CLS, INP) de cada página. pagespeed.web.dev
  • Peso total de cada página — abrir DevTools → Network → recargar → anotar el "Transferred" total.
06

Legal y Compliance

Cosas legales mínimas que un sitio con formulario y media necesita.

  • Política de privacidad — publicada en /privacidad. Obligatoria por el formulario.
  • Banner de cookies — confirmar que NO hace falta (Cloudflare Analytics no usa cookies). Si se agrega GA4 después, sí hay que poner banner.
  • Derechos de imagen — confirmar con el cliente que tiene permiso de uso para todas las fotos/videos de trabajos publicados.
07

Setup Técnico Local

Tu máquina lista para desarrollar.

  • Node.js LTS (v20+) instalado. nodejs.org/en/download — verificar con node -v.
  • Git configurado con usuario y email correctos (git config --global user.name).
  • Editor preparado — VSCode con extensión oficial de Astro instalada.
08

Post-Deploy (para después)

Tareas que se hacen una vez que el sitio nuevo ya está online. Anotalas para no olvidarlas.

  • Google Search Console — verificar dominio y submitear el sitemap.xml. search.google.com/search-console
  • Google Business Profile — si el estudio tiene oficina física, dar de alta el perfil. Aparece en Google Maps y búsqueda local.
  • Bing Webmaster Tools — equivalente a Search Console pero para Bing/DuckDuckGo (5-10% del tráfico global).
  • Plan de backups del CMS — el contenido vive en el repo git (Decap commitea), pero confirmar que el repo de GitHub tenga branch protection.
  • Recordatorio de updates — agendarse revisar dependencias cada 3-6 meses (npm outdated).
  • Probar Open Graph — pasar las URLs por opengraph.xyz y debugger de Facebook/LinkedIn para validar previews.
